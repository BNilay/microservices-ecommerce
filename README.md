# Microservices E-Commerce Application

This project is a microservices-based e-commerce backend application developed with ASP.NET Core Web API, PostgreSQL, Docker Compose, Ocelot API Gateway, JWT authentication, and Postman.

The system is designed around separate services for customers, products, orders, payments, and API gateway routing. Each core service has its own responsibility and database.

## Project Architecture

The application consists of the following services:

| Service | Description | Port |
|---|---|---|
| API Gateway | Single entry point for clients, built with Ocelot | 5000 |
| CustomerService | Handles customer registration, login, and customer data | 5001 |
| ProductService | Handles product CRUD operations and stock management | 5002 |
| OrderService | Handles order creation, order status, stock coordination, and payment integration | 5003 |
| PaymentService | Handles mock payment creation, payment status, and refund operations | 5004 |

Each main service uses its own PostgreSQL database:

| Database | Port |
|---|---|
| customer-db | 5433 |
| product-db | 5434 |
| order-db | 5435 |
| payment-db | 5436 |

## Main Features

### CustomerService

CustomerService is responsible for customer-related operations.

Main features:

- Customer registration
- Customer login
- JWT token generation
- Customer detail retrieval

Authentication tokens are generated by CustomerService and validated by the API Gateway.

### ProductService

ProductService manages product data and stock operations.

Main features:

- List products
- Get product by id
- Create product
- Update product
- Delete product
- Increase or decrease product stock

Product stock is updated by OrderService during order and cancellation flows.

### OrderService

OrderService manages the order lifecycle.

Main features:

- Create order
- List all orders
- Get order by id
- Get orders by customer id
- Update order status
- Delete order
- Check customer existence through CustomerService
- Check product information and stock through ProductService
- Call PaymentService during order creation

When an order is created, OrderService checks the customer, checks product stock, calculates the total amount, decreases stock, and calls PaymentService.

If payment is successful, the order becomes `Confirmed`.

If payment fails, the order becomes `Cancelled` and the stock is restored.

### PaymentService

PaymentService handles a mock payment flow.

This project does not use a real bank or payment provider. Instead, PaymentService simulates payment behavior.

Main features:

- Create payment
- Return successful or failed payment result
- Get payment by id
- Refund completed payment

Payment statuses:

| Status | Value |
|---|---|
| Pending | 0 |
| Completed | 1 |
| Failed | 2 |
| Refunded | 3 |

### API Gateway

The API Gateway is built with Ocelot. It provides a single entry point for external requests.

Instead of accessing services directly from different ports, clients can use:

```text
http://localhost:5000
```

Example routes:

| Gateway Route | Target Service |
|---|---|
| `/api/products` | ProductService |
| `/api/orders` | OrderService |
| `/api/payments` | PaymentService |
| `/api/customers/{id}` | CustomerService |
| `/api/auth/login` | CustomerService |
| `/api/auth/register` | CustomerService |

JWT authentication is applied at the gateway level for protected routes such as `/api/orders`.

## Order and Payment Flow

The main business flow works as follows:

```text
1. User creates an order through API Gateway.
2. OrderService checks whether the customer exists.
3. OrderService checks product information and stock.
4. OrderService creates the order as Pending.
5. Product stock is decreased.
6. OrderService calls PaymentService.
7. If payment is Completed, order status becomes Confirmed.
8. If payment is Failed, order status becomes Cancelled.
9. If order is Cancelled, stock is restored.
```

This flow demonstrates communication between OrderService, CustomerService, ProductService, and PaymentService.

## JWT Authentication Flow

The authentication flow works as follows:

```text
1. User registers through /api/auth/register.
2. User logs in through /api/auth/login.
3. CustomerService generates a JWT token.
4. Client sends the token in the Authorization header.
5. API Gateway validates the token.
6. If the token is valid, the request is forwarded to the target service.
7. If the token is missing or invalid, API Gateway returns 401 Unauthorized.
```

Example authorization header:

```text
Authorization: Bearer <token>
```

## Authentication and Authorization

Authentication means verifying who the user is.

In this project, authentication is handled by CustomerService. When a user logs in successfully through `/api/auth/login`, CustomerService generates a JWT token.

Authorization means deciding whether an authenticated user can access a specific endpoint.

In this project, authorization is handled at the API Gateway level. The `/api/orders` routes are protected with JWT. If a request does not include a valid token, API Gateway returns `401 Unauthorized`.

The client does not need to manually copy the token for every request. In the Postman collection, the login request automatically saves the token into the environment. Protected endpoints use this token with:

```text
Authorization: Bearer {{token}}
```

Current authorization design:

```text
Public routes:
- /api/auth/register
- /api/auth/login
- /api/products
- /api/payments
- /api/customers

Protected routes:
- /api/orders
```

Role-based authorization is not implemented yet. This can be added as a future improvement by assigning roles such as `Admin` and `Customer` to users.

## Running the Project

Make sure Docker Desktop is running.

From the root project folder, run:

```bash
docker compose up --build
```

Or run the containers in the background:

```bash
docker compose up -d --build
```

After the containers start, the services will be available on the following ports:

```text
API Gateway:     http://localhost:5000
CustomerService: http://localhost:5001
ProductService:  http://localhost:5002
OrderService:    http://localhost:5003
PaymentService:  http://localhost:5004
```

## Postman Collection

This project includes a Postman collection and environment file for testing the API.

Files:

```text
microservices-ecommerce.postman_collection.json
microservices-ecommerce-local.postman_environment.json
```

The Postman environment includes the following variables:

| Variable | Description |
|---|---|
| `baseUrl` | API Gateway base URL |
| `token` | JWT token saved after login |
| `customerId` | Customer id used in test requests |
| `productId` | Product id used in test requests |
| `orderId` | Order id saved after creating an order |
| `paymentId` | Payment id saved after creating a payment |

The login request automatically saves the JWT token into the environment:

```javascript
const response = pm.response.json();

pm.environment.set("token", response.token);
```

Protected requests use the token automatically:

```text
Bearer {{token}}
```

```

  "email": "test@example.com",
  "password": "123456"
}
```

Expected result:

```text
JWT token is generated and saved automatically in Postman environment.
```

### 3. Access protected orders endpoint without token

```http
GET http://localhost:5000/api/orders
```

Expected result:

```text
401 Unauthorized
```

### 4. Access protected orders endpoint with token

```http
GET http://localhost:5000/api/orders
Authorization: Bearer <token>
```

Expected result:

```text
Orders list is returned.
```

### 5. Create an order

```http
POST http://localhost:5000/api/orders
Authorization: Bearer <token>
```

Example body:

```json
{
  "customerId": 5,
  "paymentMethod": 0,
  "items": [
    {
      "productId": 5,
      "quantity": 1
    }
  ]
}
```

Expected behavior:

- If payment succeeds, order status becomes `Confirmed`.
- If payment fails, order status becomes `Cancelled`.
- If payment fails, product stock is restored.

## Multiple Product Order Example

The system supports multiple products in a single order.

Example:

```json
{
  "customerId": 5,
  "paymentMethod": 0,
  "items": [
    {
      "productId": 5,
      "quantity": 1
    },
    {
      "productId": 6,
      "quantity": 2
    }
  ]
}
```

The total amount is calculated by OrderService using product prices from ProductService.

## Notes

- PaymentService uses mock payment logic. No real payment provider is integrated.

- This project is designed as a learning-oriented microservices e-commerce backend.
